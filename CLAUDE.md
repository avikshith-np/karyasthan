# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Karyasthan ("caretaker" in Malayalam) is a WhatsApp group chat AI agent that acts as a real person in group conversations. It uses Baileys (WhatsApp Web API) for messaging and an LLM for response generation. Core philosophy: be a participant, not a servant — respond selectively, be opinionated, remember people, build relationships.

The persona is **fully configurable** — each install generates its own. `npm run setup` asks a few questions (name, age, region, language, vibe) and writes the bot's identity. The original author's own instance is, for example, a 28-year-old project manager from Kerala who speaks Manglish — but that is just one persona, not the project's definition.

### Persona files

The persona lives in two gitignored files under `src/personality/`, both generated by the setup wizard:

- **`persona.json`** — structured fields the *code* reads: `name`, `displayName`, `aliases[]` (names the bot answers to), `voiceDescriptor`, `region`, `language`, `vibe`. Loaded via `getPersona()` in `src/utils/config.js`.
- **`identity.md`** — free-form prose injected verbatim into the LLM system prompt.

Committed references (used as fallbacks if the real files are missing, and as the template the wizard renders from): `persona.example.json`, `identity.example.md`, `identity.template.md`. Never hardcode a persona name or language in code — read it from `getPersona()`.

## Commands

```bash

npm run dev            # Run with pretty-printed logs (pino-pretty)
npm run migrate        # Run database migrations
npm run test:llm       # Test LLM connection with sample messages
npm run test:decision  # Test decision engine with mock messages (10 iterations each)
npm run inspect        # Inspect/debug memory database content

```

Set `DRY_RUN=true` in `.env` to log responses instead of sending them to WhatsApp.

## Architecture

The message processing pipeline flows through these stages:

1. **Memory extraction** (`src/memory/learningPipeline.js`) — async, non-blocking. Fast heuristics on every message; LLM batch analysis every 100 messages per group.
2. **Decision engine** (`src/brain/decisionEngine.js`) — multiplicative 6-factor probability model (mention, question, humor, momentum, recency, BS detection). Direct mentions bypass probabilistic scoring at 95% confidence.
3. **Audio transcription** (`src/brain/transcribe.js`) — downloads voice messages via Baileys and transcribes using OpenRouter or native Gemini API. Transliterates non-English audio to English script.
4. **LLM generation** (`src/brain/contextBuilder.js` + `src/brain/llm.js`) — builds system prompt from identity doc + group context + people profiles + memories, then calls LLM. Multi-provider: OpenRouter (primary), Anthropic, OpenAI, Gemini, Zhipu GLM, Ollama (local fallback).
5. **Post-processing** (`src/brain/postProcess.js`) — strips AI artifacts, markdown, splits long messages.
6. **Humanization** (`src/behavior/timing.js`) — context-aware response delays, typing indicator simulation (refreshed every 7s), delayed read receipts.

Orchestrated by `src/index.js` which is the main event loop.

## Key Modules

- **`src/whatsapp/`** — Baileys socket connection, event handlers (message upsert, group updates), send actions (text, reactions, typing, read receipts, presence)
- **`src/memory/`** — SQLite (better-sqlite3) storage. Schema in `schema.sql`. Tables: messages (with FTS), people, groups, nicknames, relationships, slang, memories
- **`src/behavior/`** — Rate limiter, timing, warmup schedule for new deployments
- **`src/personality/identity.md`** — Full character definition. `systemPrompt.js` dynamically assembles the prompt from identity + learned context
- **`src/utils/`** — Config (env loading/validation), logger (Pino), JID parsing, path safety
- **`src/dashboard/`** — Embedded Fastify web UI (read-only v1). Starts alongside the bot when `DASHBOARD_ENABLED=true`. Reuses `db`, `config`, `logger`, and the live Baileys socket from the main process. Routes under `src/dashboard/routes/`; static SPA in `src/dashboard/public/` (htmx + Alpine + Pico, no build step).

## Important Patterns

- **JID format**: WhatsApp uses `sender@s.whatsapp.net` (individuals) and `groupid@g.us` (groups) — JID utilities in `src/utils/jidUtils.js`
- **Fire-and-forget learning**: Heuristics and batch analysis never block the response pipeline
- **Warmup system**: Days 1-3 are reactions-only; text responses gradually increase over ~3 weeks. State stored in `data/warmup.json`
- **LLM skip signal**: If LLM returns `[SKIP]` or `SKIP`, interpreted as intentional decision not to respond
- **Cooldowns are in-memory**: Burst cooldowns reset on restart
- **ESM modules**: Project uses `"type": "module"` — all imports are ES module style

## Configuration

All config via `.env` (see `.env.example`). Key variables:

- **LLM**: `LLM_PROVIDER` (openrouter/anthropic/openai/gemini/ollama), `LLM_MODEL`, `LLM_API_KEY`, `OPENROUTER_API_KEY`, `GEMINI_API_KEY`, `GLM_API_KEY`, `LLM_FALLBACK_PROVIDER`, `LLM_FALLBACK_MODEL`, `LLM_MAX_TOKENS`, `LLM_TEMPERATURE`, `LLM_MONTHLY_BUDGET_USD`, `TRANSCRIPTION_MODEL`
- **WhatsApp**: `WA_PHONE_NUMBER`
- **Behavior**: `RESPONSE_RATE` (base probability, default 0.15), `MAX_RESPONSE_LENGTH`, `SLEEP_START_HOUR`/`SLEEP_END_HOUR`, `TIMEZONE`
- **Rate limits**: `MAX_PER_GROUP_HOUR`, `MAX_GLOBAL_HOUR`
- **Paths**: `DB_PATH`, `AUTH_PATH`, `LOG_LEVEL`
- **Toggles**: `WARMUP_ENABLED`, `DRY_RUN`
- **Dashboard**: `DASHBOARD_ENABLED`, `DASHBOARD_HOST` (default `127.0.0.1`), `DASHBOARD_PORT` (default `7070`), `DASHBOARD_USER`, `DASHBOARD_PASS_HASH` (bcrypt), `DASHBOARD_COOKIE_SECRET`, `DASHBOARD_ALLOW_CIDRS` (comma-sep CIDRs), `DASHBOARD_READONLY` (v1 hard-defaults to `true`)

WhatsApp auth credentials stored in `data/auth_info_baileys/`. SQLite database at `data/karyasthan.db`.

## Dashboard

Web UI for observing and steering the bot. Enable with `DASHBOARD_ENABLED=true` and set `DASHBOARD_USER` + `DASHBOARD_PASS_HASH`. Default bind is `127.0.0.1:7070` — reach it via SSH tunnel, Tailscale (100.64.0.0/10 is in the default CIDR allowlist), or local only. Not public internet.

```bash
# Generate a bcrypt hash for DASHBOARD_PASS_HASH
node -e 'import("bcryptjs").then(b=>b.default.hash("yourpass",10).then(console.log))'
```

Pages (hash-routed SPA): `#status`, `#groups` → `#group/<jid>`, `#people` → `#person/<jid>`, `#memories`, `#decisions`, `#messages` (FTS), `#bills`, `#logs` (live SSE), `#settings`, `#identity`, `#skills`, `#audit`.

### Auth

- Basic-auth (user + bcrypt pass hash)
- 30-day signed session cookie (`dash_session`) — frontend POSTs `/api/login` on boot to convert basic-auth into the cookie
- Stricter rate-limit on `/api/login` (20/min) vs global (300/min)
- `DASHBOARD_READONLY=true` blocks all write methods. Default is `true` — flip to `false` once you're ready to edit state
- CSRF: when `DASHBOARD_READONLY=false`, every write must carry `X-CSRF-Token` header matching the `csrf_token` cookie issued at login

### Writes

- `PATCH /api/config` — live-tune allow-listed keys (`responseRate`, `dryRun`, `llm.temperature`, `llm.maxTokens`, `qualityGate.*`, rate caps, feature toggles). In-memory only; a pm2 restart reverts unless you also edit `.env`
- `POST/PATCH/DELETE /api/memories` — memory CRUD
- `PATCH /api/people/:jid` — edit `real_name`, `summary`, `traits[]`, `interests[]`
- `PATCH /api/groups/:jid` — edit `vibe`, `language`
- `POST /api/groups/:jid/mute` (`{durationMinutes?, reason?}`) and `/unmute` — mute blacklists the group from the response pipeline entirely; learning continues
- `PUT /api/identity` — writes `src/personality/identity.md` and hot-reloads into the running bot
- `POST /api/skills/:name/toggle` — enable/disable a loaded skill at runtime
- `POST /api/maintenance/{memory,bills}` — run cleanup jobs on demand
- `GET /api/audit` — every dashboard write is recorded in `dashboard_audit`

API surface is under `/api/*`; the static SPA is served from `src/dashboard/public/`. `scripts/smoke-dashboard.js` boots the dashboard on port 7071 without Baileys for local API testing.

### Still out of scope (Phase 4+)

- `POST /api/pair/start` — re-pair WhatsApp via the UI
- `POST /api/config/persist` — write the live config back to `.env` with a timestamped backup
- Extracting decision-engine weights to a tunable table

## Deployment

A systemd unit file is provided in `karyasthan.service`. Copy to `/etc/systemd/system/` and enable with `sudo systemctl daemon-reload && sudo systemctl enable --now karyasthan`.
